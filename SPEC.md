# Contextual Subtitle Studio

## 1. 概要

本プロジェクトは、英語SRT字幕を日本語字幕へ翻訳するための、ドラマ特化型字幕翻訳ツールである。

通常の機械翻訳やLLM翻訳では、以下の問題が生じやすい。

1. 固有名詞の表記がぶれる
2. 登場人物間の上下関係が崩れる
3. 性別に基づく不自然な女言葉・男言葉が混入する
4. 作品固有の称号、身分、呼称、関係性が反映されない
5. 行単位翻訳により、前後文脈を踏まえた自然な字幕にならない

本ツールは、単なるSRT翻訳ツールではなく、TMDb/IMDb等の英語情報と、中国語圏のドラマ情報ページから得られる人物名・役名・関係・身分・あらすじを利用して、翻訳前に作品用の人物辞書・用語表・敬語ルールを生成する。

翻訳にはDeepSeek、MiniMax、OpenAI互換API、ローカルLLM等を利用できるようにし、特定のLLMには依存しない。

## 2. 基本方針

### 2.1 本ツールの中心目的

本ツールの目的は、英語SRTを自然な日本語字幕に翻訳することである。

ただし、一般的な翻訳ツールとは異なり、以下を重視する。

* 作品ごとの固有名詞の統一
* 登場人物の身分・上下関係の反映
* 過剰な女言葉・男言葉の回避
* 中性的で簡潔な日本語字幕調
* 前後文脈を踏まえた翻訳
* 問題行の検出と再翻訳
* 人間による最終確認・修正を前提とした半自動処理

### 2.2 字幕文体の基本方針

日本語字幕では、性別に基づく強い語尾づけを原則として避ける。

避ける表現例:

* 「〜だわ」
* 「〜かしら」
* 「〜なのよ」
* 「〜だぜ」
* 「〜じゃねえか」
* 「お前、何してんだよ」

基本方針:

* 中性的で簡潔な日本語字幕調にする
* 性別よりも、身分、親密度、敵対関係、年齢差、職位差を重視する
* 皇帝、王、上官、師、親などの上位者への発話では、ため口を避ける
* 敬語を使いすぎて字幕が冗長になることは避ける
* 1字幕あたりの文字数を増やしすぎない

### 2.3 自動化の範囲

本ツールは完全自動翻訳を目指さない。

LLMによる翻訳・検査・再翻訳を行うが、最終的にはユーザーが以下を確認できるようにする。

* 人物辞書
* 用語表
* 敬語ルール
* 翻訳結果
* 問題行レポート
* 再翻訳候補

## 3. 想定ユーザー

主なユーザーは以下である。

* 海外ドラマ、特に中国ドラマ・韓国ドラマ等の英語SRTを日本語化したい個人ユーザー
* 機械翻訳字幕の固有名詞ぶれや口調崩れを修正したいユーザー
* LLM APIを使って字幕翻訳ワークフローを半自動化したいユーザー
* SRTファイルを保持したまま自然な日本語字幕を作りたいユーザー

* 問題行レポート
* 再翻訳候補レポート

## 5. 全体ワークフロー

本ツールは以下の流れで動作する。

```text
1. 英語SRTを読み込む
2. TMDb/IMDb等から英語側の作品情報を取得する
3. 中国語圏のドラマ情報ページから中国語側の作品情報を取得する
4. LLMが英語情報と中国語情報を照合する
5. 人物辞書・用語表・敬語ルールを生成する
6. ユーザーが人物辞書を確認・修正する
7. SRTをチャンク単位に分割する
8. LLMで日本語字幕に翻訳する
9. 翻訳結果を検査する
10. 問題行を抽出する
11. 問題行のみ再翻訳する
12. 最終SRTを出力する
```

## 6. 作品情報インポート機能

### 6.1 基本方針

本ツールでは、大量クローリングや定期巡回は行わない。

作品ごとに、ユーザーが明示的に実行した場合のみ、LLMが検索・ページ閲覧を行い、人物辞書の作成に必要な情報を抽出する。

この機能は「クローラー」ではなく、「作品情報インポート」または「LLMリサーチ補助」と位置づける。

### 6.2 対象となる情報源

英語側:

* TMDb（API経由、優先）
* IMDb（補助）
* Wikipedia
* 公式配信サイトの英語ページ
* その他、ユーザー指定URL

中国語側:

* 豆瓣电影 / 豆瓣剧集
* 电视猫 / TVMao
* 猫眼专业版
* 百度百科
* 抖音百科 / 快懂百科
* 爱奇艺
* 腾讯视频
* 优酷
* 芒果TV
* 中文维基百科
* その他、ユーザー指定URL

### 6.3 取得する情報

作品情報:

* 英語タイトル
* 中国語タイトル
* 別名
* 原題
* 放送・配信年
* ジャンル
* 話数
* あらすじ
* 時代設定
* 主要な勢力・組織

人物情報:

* 英語名
* 中国語名
* 日本語表記案
* 俳優名
* 役名
* 別名
* 称号
* 身分
* 所属
* 人物関係
* 呼称
* 敬語レベル
* 翻訳上の注意点

### 6.4 取得方式

初期実装では、以下の3方式を想定する。

#### 方式A: URL指定型

ユーザーがTMDbや中国語DBのURLを直接入力する。

LLMは指定されたページを読み、必要情報を抽出する。

#### 方式B: 検索補助型

ユーザーが作品名を入力する。

LLMが検索し、候補ページを提示する。

ユーザーが採用するページを選び、そのページだけを読み込む。

#### 方式C: 手動貼り付け型

Web取得が難しい場合、ユーザーがページ本文やキャスト表をコピーして貼り付ける。

LLMは貼り付けられたテキストから人物表を生成する。

### 6.5 キャッシュ方針

同じURLに何度もアクセスしないよう、取得結果はローカルにキャッシュする。

キャッシュには以下を保存する。

* URL
* 取得日時
* ページタイトル
* 抽出本文
* 抽出結果
* 使用したLLMモデル
* ユーザーによる修正履歴

## 7. 人物辞書

### 7.1 目的

人物辞書は、翻訳時の固有名詞、呼称、敬語、口調を安定させるために使用する。

### 7.2 基本構造

人物辞書はJSONまたはCSVで保存する。

例:

```json
{
  "characters": [
    {
      "id": "chu_qiao",
      "english_name": "Chu Qiao",
      "chinese_name": "楚乔",
      "japanese_name": "楚喬",
      "aliases": ["Qiao", "Xing'er"],
      "actor": "黄杨钿甜",
      "role": "主人公",
      "status": "奴籍少女、のちに武将",
      "gender": "female",
      "default_register": "neutral",
      "speech_style": "中性的で簡潔な字幕調",
      "notes": "過剰な女言葉は避ける"
    }
  ]
}
```

### 7.3 重要フィールド

必須に近いフィールド:

* `id`
* `english_name`
* `chinese_name`
* `japanese_name`
* `aliases`
* `role`
* `status`
* `default_register`
* `speech_style`
* `notes`

任意フィールド:

* `actor`
* `gender`
* `age_group`
* `organization`
* `rank`
* `relationship_notes`
* `source_urls`
* `confidence`

### 7.4 人物関係

人物間の関係は、別オブジェクトとして保持する。

例:

```json
{
  "relationships": [
    {
      "from": "minister_shen",
      "to": "emperor_xiao",
      "relationship": "臣下から皇帝",
      "register": "formal",
      "must_use": ["陛下", "ございます", "いたします"],
      "avoid": ["お前", "君", "だよ", "じゃん"],
      "notes": "皇帝に対してため口不可"
    }
  ]
}
```

## 8. 用語表

### 8.1 目的

用語表は、作品内の固有名詞、称号、地名、組織名、武器名、技名などを統一するために使用する。

### 8.2 用語表の例

```json
{
  "glossary": [
    {
      "source": "His Majesty",
      "target": "陛下",
      "type": "title",
      "notes": "皇帝への呼称"
    },
    {
      "source": "Yanbei",
      "target": "燕北",
      "type": "place",
      "notes": "地名"
    },
    {
      "source": "Zhuge Yue",
      "target": "諸葛玥",
      "type": "character",
      "notes": "人物名"
    }
  ]
}
```

### 8.3 表記方針

日本語表記は、初期状態では以下を基本とする。

* 簡体字は原則として日本語漢字に寄せる
* ただし、一般に中国名として認識される場合は簡体字・繁体字風表記も許容する
* ユーザーが最終的に表記を確定する
* 一度確定した表記は翻訳中に変更しない

## 9. 敬語・口調ルール

### 9.1 基本方針

性別に基づく女言葉・男言葉の指定は最小限にする。

優先順位は以下とする。

1. 身分差
2. 上下関係
3. 親密度
4. 敵対関係
5. 場面の緊張度
6. 性別

### 9.2 敬語レベル

初期実装では、以下の敬語レベルを用意する。

```text
plain        : くだけた普通体
neutral      : 中性的・簡潔な字幕調
polite       : 丁寧語
formal       : 身分差を反映した丁寧な文体
honorific    : 強い敬意を含む文体
commanding   : 上位者・命令者の文体
hostile      : 敵対的な文体
intimate     : 親密な文体
```

### 9.3 禁止表現リスト

初期状態では、以下のような表現を過剰な性別語尾として検出対象にする。

```text
だわ
かしら
なのよ
なのね
ですわ
だぜ
だぞ
じゃねえ
てめえ
```

ただし、作品やキャラクターによっては意図的に許可できるようにする。

## 10. LLMプロバイダ設計

### 10.1 基本方針

本ツールは特定のLLMに依存しない。

DeepSeek、MiniMax、OpenAI互換API、OpenRouter、Ollama、LM Studio、独自ゲートウェイなどを切り替え可能にする。

### 10.2 モデル選択

モデルは工程ごとに選べるようにする。

工程:

* 作品情報抽出
* 人物辞書生成
* 用語表生成
* スタイルガイド生成
* 字幕翻訳
* 翻訳検査
* 問題行再翻訳

設定例:

```yaml
models:
  metadata_model: minimax-m3
  glossary_model: minimax-m3
  style_model: minimax-m3
  translator_model: deepseek-v4-pro
  reviewer_model: deepseek-v4-pro
  repair_model: deepseek-v4-pro
```

### 10.3 初期対応プロバイダ

初期対応候補:

* DeepSeek
* MiniMax
* OpenAI互換API
* OpenRouter
* LM Studio
* Ollama
* 独自OpenAI互換ゲートウェイ

### 10.4 プロバイダ設定

例:

```yaml
providers:
  deepseek:
    type: openai_compatible
    base_url: "https://api.deepseek.com"
    api_key_env: "DEEPSEEK_API_KEY"

  minimax:
    type: openai_compatible
    base_url: "https://api.minimax.io/v1"
    api_key_env: "MINIMAX_API_KEY"

  local:
    type: openai_compatible
    base_url: "http://localhost:1234/v1"
    api_key: "dummy"
```

### 10.5 モデルプロファイル

モデルごとの差を吸収するため、モデルプロファイルを持つ。

例:

```yaml
model_profiles:
  minimax-m3:
    max_context_tokens: 1000000
    supports_json: true
    strip_thinking_tags: true
    recommended_batch_size: 80

  deepseek-v4-pro:
    supports_json: true
    strip_thinking_tags: true
    recommended_batch_size: 50

    supports_json: false
    strip_thinking_tags: true
```

## 11. 翻訳パイプライン

### 11.1 SRT解析

SRTを以下の構造に分解する。

```json
{
  "index": 1,
  "start": "00:00:01,000",
  "end": "00:00:03,000",
  "text": "Original subtitle text"
}
```

### 11.2 チャンク分割

字幕は一定数ごとにチャンク分割する。

ただし、単純な行数だけでなく、以下を考慮する。

* 場面転換
* 長い無音区間
* 会話のまとまり
* 登場人物の切り替わり
* LLMのコンテキスト長

### 11.3 翻訳時に渡す情報

各チャンク翻訳時には、以下をLLMに渡す。

* 翻訳対象SRTチャンク
* 直前の要約
* 直後の軽い文脈
* 作品スタイルガイド
* 関連人物辞書
* 関連用語表
* 敬語・口調ルール
* 出力形式指定

### 11.4 翻訳プロンプトの基本方針

翻訳プロンプトには以下を含める。

```text
あなたは日本語字幕翻訳者です。
英語SRTを自然な日本語SRTに翻訳してください。

規則:
- SRT番号とタイムコードは変更しない。
- 出力は翻訳済みSRTのみ。
- 固有名詞は用語表に必ず従う。
- 性別に基づく過剰な女言葉・男言葉は避ける。
- 基本は中性的で簡潔な字幕調にする。
- 身分差、上下関係、親密度、敵対関係は反映する。
- 皇帝・王・上官・師への発話では、ため口を避ける。
- 意味を補いすぎない。
- 1字幕あたりの文字数を増やしすぎない。
```

### 11.5 出力形式

初期実装では、翻訳結果はSRT形式で受け取る。

将来的には、より安全なJSON形式も検討する。

例:

```json
[
  {
    "index": 1,
    "start": "00:00:01,000",
    "end": "00:00:03,000",
    "translation": "日本語字幕"
  }
]
```

## 12. 翻訳検査

### 12.1 検査項目

翻訳後、以下を検査する。

* SRT番号が保持されているか
* タイムコードが保持されているか
* 未翻訳の英語が残っていないか
* 固有名詞が用語表に従っているか
* 同一人物の表記がぶれていないか
* 皇帝・王・上官・師へのため口がないか
* 過剰な女言葉・男言葉が混入していないか
* 字幕として長すぎないか
* 内容の脱落がないか
* 余計な説明が追加されていないか
* LLMの説明文や思考タグが混入していないか

### 12.2 検査方法

検査は以下を組み合わせる。

1. ルールベース検査
2. 用語表照合
3. SRT構造検査
4. LLMレビュー
5. ユーザー確認

### 12.3 問題行レポート

検査結果はJSONで保存する。

例:

```json
{
  "issues": [
    {
      "index": 42,
      "type": "register_error",
      "severity": "high",
      "message": "臣下が皇帝に対してくだけた口調を使用している可能性があります",
      "source_text": "I don't know.",
      "translation": "分からない",
      "suggestion": "分かりません"
    }
  ]
}
```

## 13. 再翻訳

### 13.1 基本方針

全体を再翻訳するのではなく、問題が検出された行またはチャンクのみ再翻訳する。

### 13.2 再翻訳時に渡す情報

再翻訳時には以下を渡す。

* 問題行
* 問題の種類
* 元の英語
* 現在の日本語訳
* 前後の字幕
* 関連人物情報
* 関連敬語ルール
* 修正方針

### 13.3 再翻訳後の確認

再翻訳後も、再度検査を行う。

複数候補を出して、ユーザーが選べるようにすることも検討する。

## 17. 非目標

初期段階では、以下は対象外とする。

* 動画からの音声認識
* 字幕タイミングの自動生成
* 動画への字幕焼き込み
* 完全自動の大量Webクローリング
* 動画配信サイトからの動画取得
* DRM回避
* 著作権保護コンテンツの不正取得
* 商用字幕制作向けの完全自動納品
* 翻訳著作権・配布権の保証
* すべての言語ペアへの対応

## 18. 法的・倫理的配慮

### 18.1 Web情報取得

本ツールは、大量クローリングや定期巡回を行わない。

ユーザーが明示的に指定した作品について、数ページ程度の情報を取得し、人物辞書作成の補助に利用する。

設計上、以下を守る。

* ユーザー操作時のみ取得する
* 同一URLへの反復アクセスを避ける
* 取得結果をローカルキャッシュする
* ログイン回避やアクセス制限突破は行わない
* bot対策を回避しない
* 取得失敗時は手動貼り付けに切り替える
* 画像・動画・大量ページを取得しない

### 18.2 字幕・動画の扱い

本ツールは、ユーザーが正当に保持しているSRTファイルを翻訳するための補助ツールである。

本ツールは以下を行わない。

* 動画の違法ダウンロード
* DRM回避
* 海賊版配布
* 翻訳字幕の無断公開支援

## 19. セキュリティ

### 19.1 APIキー管理

APIキーは平文でリポジトリに保存しない。

対応方式:
* 環境変数

### 19.2 ローカルデータ

プロジェクトデータはローカルに保存する。

外部APIに送信される可能性がある情報:

* SRT本文
* 人物辞書
* 用語表
* 作品メタデータ
* 翻訳指示

ユーザーには、APIに送信される情報を明示する。

## 22. リスク

### 22.1 翻訳品質リスク

* LLMがSRT形式を壊す
* 固有名詞を勝手に訳す
* 敬語が過剰または不足する
* 文脈を誤解する
* 作品情報ページから誤情報を拾う
* 同名人物を混同する

対策:

* JSONまたは厳格なSRT出力指定
* ルールベース検査
* 用語表照合
* 人物辞書の人間確認
* 問題行再翻訳
* 翻訳ログ保存

### 22.2 Web情報取得リスク

* ページ構造変更
* アクセス制限
* bot対策
* 情報の古さ
* 作品ページの取り違え

対策:

* URL指定型を基本にする
* 検索結果はユーザーが確認する
* 手動貼り付けに対応する
* 情報源URLを保存する
* 信頼度を付ける

### 22.3 モデル依存リスク

* モデルにより出力形式が不安定
* JSONが壊れる
* thinkingタグが混入する
* 長文脈で指示が薄れる
* コストが高くなる

対策:

* モデルプロファイル
* 出力後処理
* SRT構造検査
* チャンクサイズ調整
* 工程ごとのモデル選択

## 24. 初期開発方針

初期開発では、WebインポートやGUIよりも、まず以下を優先する。

1. SRTを壊さず翻訳できること
2. 人物辞書を翻訳に反映できること
3. 固有名詞ぶれを検出できること
4. 過剰な女言葉・男言葉を抑制できること
5. 上下関係ルールをある程度反映できること
6. モデルを切り替えられること

最初から完全なドラマ情報自動取得を目指さず、まずは手動で作成した人物辞書を使って、翻訳品質がどの程度改善するかを検証する。

その後、TMDbや中国語DBの情報をLLMで読み取り、人物辞書を半自動生成する機能を追加する。
